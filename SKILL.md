# Novel-Craft：网文手法拆解与创作 skill

## 这个 skill 在验证什么

这不只是一个写作工具。它是一条链的测试床：

**差异 → 为什么有效 → 机制卡片 → 判断（何时用）→ 反馈更新**

小说场景的特殊价值在于：**反馈极其干净**。评判官就是用户本人，坐在这里，即时、低噪、能说出"为什么"。一段写得假不假、有没有 AI 味，用户一眼就能定，并且能解释原因。这让"带因果的失败数据"几乎零成本产生——而这正是整条链最贵、最难拿的原料。

所以运行这个 skill 时，目标永远是：**让这条链真正动起来、能自我迭代**，而不只是交付一段文字。

---

## 三层结构（这个 skill 可迁移的关键）

这个 skill 严格区分三种东西，绝不能揉成一坨。揉在一起，skill 就被绑死在一本书上、失去普遍性；分开，才能写一百本书都复用。

```
┌─────────────────────────────────────────────────────────┐
│ 1. SKILL.md —— 纯方法，永久不变                          │
│    三层拆解、改写回路、防 AI 味、记忆管理流程            │
│    迁移到任何一本书，一个字都不用改。                    │
├─────────────────────────────────────────────────────────┤
│ 2. 机制卡片库 —— 跨书复用的手法资产，只增不弃            │
│    从任何作品拆出来的手法卡，独立于任何一本书存在。      │
│    写第一本书攒 20 张，写第二本直接继承这 20 张、         │
│    再拆出新的，越攒越厚。这是真正的护城河。              │
├─────────────────────────────────────────────────────────┤
│ 3. 故事圣经 —— 单本书的专属记忆，跟着书走，不迁移        │
│    人物、伏笔、设定、时间线。每本书一份，互不相干。      │
└─────────────────────────────────────────────────────────┘
```

**为什么这样分：** 这正是 EI Framework 的核心区分——行业 Skill 只能产生同类 Skill，Meta Skill 能产生无限 Skill。机制卡片库就是 Meta Skill 资产：它不属于任何一本书，因此能注入每一本书。一张"信息缺口钩子"的卡片，从 A 书拆出来，照样能用在毫不相干的 B 书上。

**推荐的文件/目录结构（建议放进 Git 仓库管理）：**

```
novel-craft/
├── SKILL.md                    # 方法，不变（第 1 层）
├── mechanism-library/          # 机制卡片库，跨书累积（第 2 层）
│   ├── hooks.md                #   钩子类卡片
│   ├── pacing.md               #   节奏/断章类卡片
│   ├── dialogue.md             #   对话/人物类卡片
│   ├── ai-flavor-blacklist.md  #   "AI味失败清单"，踩过的坑
│   └── ...
└── books/                      # 每本书一个文件夹（第 3 层）
    ├── book-A/
    │   ├── story-bible.md       #   故事圣经
    │   ├── chapter-outline.md   #   章纲
    │   ├── recap.md             #   滚动前情提要
    │   └── chapters/            #   正文
    └── book-B/
        └── ...
```

写新书时：`SKILL.md` 不动，`mechanism-library/` 整个继承，只在 `books/` 下新建一个文件夹。这就是"普遍性 + 可迁移"的落地形态。

## 两个模式

### 模式 A：学习模式（拆解别人的作品）

用户喂进一篇小说（或一个片段）。**不要泛泛总结，要逼出机制。**

铁律：**拆"为什么有效"和"作者克制了什么没写"，不拆"是什么"。**
- ❌ "这本节奏快、悬念强、文笔好" —— 这是贴标签，没有任何可复用价值
- ✅ "第一句先抛结果不给原因，制造信息缺口；接下来三段都在拖延揭晓，每段结尾再加一个新疑问" —— 这是机制

对每个值得拆的片段，做三层拆解：

1. **表层（具体、可测量的变量）**
   开场怎么起？句子长短的分布？对话和叙述的比例？视角是谁？章节在哪里断、为什么断在那？哪个词用得反常但有效？

2. **中层（为什么有效——机制）**
   这个钩子制造了什么信息缺口？这段克制（没写出来的部分）让读者自己补全了什么？这个转折为什么读者会信？删掉这句会损失什么？

3. **底层（对应哪条人性规律）**
   信息缺口 → 大脑优先处理未完成信息
   身份投射 → 读者把自己代入主角
   损失厌恶 / 即时满足 / 社会认同 / 稀缺 …
   （到人性层就停，不要硬拗。）

**对照，而不是孤立。** 如果手上有多篇，对同一种场景（摊牌、初遇、打脸、转折、断章）做横向对比：武侠怎么处理、悬疑怎么处理、言情怎么处理。对照能逼出"风格的变量"，孤立分析只能得到"风格的样本"。

每拆出一条有价值的手法，写成一张**机制卡片**（格式见下）。

### 模式 B：创作模式（用学到的机制写）

**关键警告：分析能力 ≠ 写作能力。** 能把一本书拆得头头是道，写出来照样一股 AI 味——因为"看懂别人怎么失控"和"自己敢失控"是两回事。所以绝不能"分析一堆 → 直接生成整章"，中间那步是断的。

正确的回路是**小步 + 用户评判**：

1. 拿用户的大纲 / 已有片段 + 提炼出的机制卡片
2. **只改写或新写一小段**（一个场景、几百字，不要整章）
3. 把这一段交给用户评判：哪里假？哪里有 AI 味？哪个转折没信？哪句太顺太平？
4. 用户毙掉的每一处 + 说出的原因 = 一条带 Why 的失败数据 → 更新机制卡片 / 新增"AI味失败清单"
5. 带着更新后的认知，重写这一段或写下一段
6. 循环

这个回路本身就是这个 skill 最值钱的东西。不要跳过用户评判直接往下冲。

## AI 味从哪来（写的时候盯死这三条）

拆解能压下表层结构的 AI 味，但下面这三条是底层倾向，会在"写"的环节反复冒出来，必须主动对抗：

1. **安全无菌感**：AI 倾向把每个情绪解释清楚、每个角色写得通情达理、回避真正的尖锐和失控。好小说里那种说不清的恶意、不讲道理的冲动、留白不解释的部分，要敢保留，不要抹平。

2. **遣词平均化**：AI 默认写"最可能的那个句子"——最顺、最稳、最不会出错，所以最没记忆点。要敢用怪一点的词、敢在该短的地方短到突兀、敢有执念。

3. **没有真正的意图**：人写一段是因为真的想说某件事；AI 写一段是因为"这里该有一段"。每写一段先问：这段到底想让读者产生什么具体反应？没有就别写。

用户的判断永远优先于这三条规则——他说哪里有 AI 味，那里就是有，照他的改。

## 机制卡片格式

```
手法名称：[一句话命名]
来源：[哪篇小说的哪个场景]
类型：[钩子 / 节奏 / 断章 / 视角 / 对话 / 留白 / 打脸 / …]
表层变量：[具体、可测量的写法]
机制：[为什么有效]
人性规律：[对应哪条]
可迁移场景：[还能用在什么地方]
失效场景：[什么时候不该用]
置信度：[假设 / 验证中 / 已验证 —— 经过用户在创作模式里实测才升级]
```

卡片不是写完就完。每次在创作模式里用了某张卡片、用户给了反馈，就回来更新它的置信度和失效场景。**没经过实测的卡片都只是假设。**

**卡片库是跨书资产（第 2 层）。** 所有卡片存进 `mechanism-library/`，独立于任何一本书。写新书时整个继承、直接调用；新书里又拆出新卡片，就追加进去。卡片库只增不弃，越厚越值钱——它是这个 skill 唯一会随时间升值的东西。

## 怎么开始（给运行这个 skill 的 Claude）

1. **先确认用户在哪个模式。** 是要拆解，还是要写，还是先拆再写。
2. **从小开始。** 一篇小说、一个场景。不要让用户先攒一个语料库——这阶段是验证闭环动没动，不是建数据库。
3. **学习模式**：拆完给出机制卡片，并主动问用户"这个拆解对不对、你读的时候是不是这种感觉"——用户的认同/反驳本身就是高信噪比反馈。
4. **创作模式**：写一小段 → 停 → 交给用户评判 → 更新 → 再走。永远不要一口气写完整章然后才问感觉。
5. **始终诚实**：如果一段自己写出来都觉得平、觉得有 AI 味，先说出来，不要假装它好。

## 模式 C：长篇记忆管理（写整本书时必须开启）

写一两个片段不需要这套。但只要是写一本完整的书，AI 的上下文窗口必然装不下全文——写到后面会忘掉前面的伏笔、人物口吻会漂、前后会矛盾。解法不是"让 AI 记住全文"，而是把记忆搬到文件里，每章只喂"写这一章必须知道的那部分"。

这正是 EI Framework 的循环：行动 → 反馈 → Memory Update → 再行动。AI 不记全文，它每次读这几份随时更新的记忆文件。

### 必须维护的记忆文件

1. **故事圣经（Story Bible）—— 最重要**
   - 人物卡：每个角色的外貌、性格、**说话习惯（口吻是防漂移的关键）**、动机、底牌、与其他人的关系
   - 世界设定：规则、地理、势力、专有名词
   - **伏笔清单**：埋在第几章、是什么、回收了没有（写"边写边想"型最容易埋了就忘，这份是救命的）
   - 时间线：发生顺序，防止前后矛盾

2. **章纲**：逐章一两句——这章发生了什么、留了什么钩子。全书的结构地图。

3. **前情提要（滚动）**：最近 2-3 章的稍详细概括 + **上一章结尾的原文**（保证口吻和语气接得住）。

4. **机制卡片库**：学习模式拆出来的手法卡，写的时候调用。

### 写每一章的标准流程

```
喂给 AI = 故事圣经 + 相关章纲 + 前情提要 + 这章要写什么 + 几张机制卡片
   ↓
AI 写这
